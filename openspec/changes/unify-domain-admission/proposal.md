## Why

TypeLayout and XOffset have **diverged implementations of the same logical check**: "is type T safe for byte-copy into shared/persistent memory?"

TypeLayout provides `is_local_serialization_free_v<T>`, which answers a **strict subset** of this question: `trivially_copyable(T) && !has_pointer(T)`. XOffset extends this domain with offset_ptr-based containers (XString, XVector, etc.) that are NOT trivially_copyable but ARE byte-copy safe. To admit these, XOffset **duplicates TypeLayout's P2996 reflection infrastructure** (`accept_all_members_impl`, `accept_all_bases_impl`) to build its own recursive admission gate (`DefaultPolicy::accept<T>()`).

The result: two codebases, each with its own P2996-based member traversal, performing overlapping but incompatible domain checks. Any structural change to TypeLayout's reflection (e.g. adding `SignatureMode`, changing opaque detection) requires mirroring in XOffset. The XOffset external TypeLayout copy has already diverged (it has `SignatureMode` parameter; TypeLayout main does not).

This proposal designs a unified domain admission API in TypeLayout that eliminates the duplication while preserving XOffset's ability to extend the safe domain with relocatable types.

## What Changes

### Core change: `is_byte_copy_safe_v<T>` -- recursive admission with opaque awareness

TypeLayout gains a new compile-time predicate that does what XOffset's `DefaultPolicy::accept<T>()` does today, but as a first-class TypeLayout API:

```
is_byte_copy_safe_v<T> =
  | has_opaque_signature<T>    -> !layout_traits<T>::has_pointer
  |                               && opaque_elements_safe<T>()
  | is_local_serialization_free_v<T> -> true
  | is_class && !union && !polymorphic
  |                            -> all bases byte_copy_safe
  |                               && all members byte_copy_safe
  | otherwise                  -> false
```

This is the **exact logic** that XOffset's `DefaultPolicy::accept()` implements today, lifted into TypeLayout. The key difference from `is_local_serialization_free_v<T>`:

- `is_local_serialization_free` is a **flat, whole-type check** -- if the top-level type is not trivially_copyable, it fails immediately, even if every member is individually safe.
- `is_byte_copy_safe` is a **recursive, per-member check** -- it decomposes the type via P2996 and checks each member independently. A struct containing only opaque + serialization-free members passes, even if the struct itself is not trivially_copyable.

### Supporting change: `opaque_elements_safe<T>()` customization point

Today XOffset has `opaque_element_types<T>::all_elements_safe()` to check container element types. TypeLayout needs an equivalent. The CONTAINER_RELOCATABLE and MAP_RELOCATABLE macros already embed element type signatures -- but they lack the recursive admission check on element types.

Proposed: Each RELOCATABLE container/map macro generates an `opaque_elements_safe<T>` specialization that recursively applies `is_byte_copy_safe` to each element type parameter.

### Non-changes

- `is_local_serialization_free_v<T>` is **preserved** as-is. It remains the answer to "can I memcpy this type without any library-specific knowledge?" -- the strict, C++-standard-only predicate.
- `layout_traits<T>` is unchanged.
- Opaque registration macros (TYPELAYOUT_REGISTER_OPAQUE, TYPELAYOUT_OPAQUE_TYPE_RELOCATABLE, etc.) are unchanged in their signature generation. They gain an optional `opaque_elements_safe` specialization.
- Existing tests are unaffected.

## Capabilities

### New Capabilities

- **is_byte_copy_safe<T>**: Recursive compile-time predicate. Returns true if T is safe for byte-copy given all registered opaque type guarantees. Requires P2996.
- **opaque_elements_safe<T>**: Customization point for opaque container/map templates to declare element type safety. Auto-generated by CONTAINER_RELOCATABLE and MAP_RELOCATABLE macros.

### Modified Capabilities

- **TYPELAYOUT_OPAQUE_CONTAINER_RELOCATABLE**: Additionally generates `opaque_elements_safe` specialization.
- **TYPELAYOUT_OPAQUE_MAP_RELOCATABLE**: Additionally generates `opaque_elements_safe` specialization.

## Impact

### TypeLayout files affected

| File | Change |
|------|--------|
| `layout_traits.hpp` (or new `admission.hpp`) | Add `is_byte_copy_safe<T>`, `opaque_elements_safe<T>`, recursive P2996 traversal |
| `opaque.hpp` | CONTAINER_RELOCATABLE and MAP_RELOCATABLE macros generate `opaque_elements_safe` |
| `typelayout.hpp` | Include new header |
| New test: `test_byte_copy_safe.cpp` | Core test for recursive admission |

### XOffset impact (downstream consumer)

After this change, XOffset's `DefaultPolicy::accept<T>()` becomes:

```cpp
struct DefaultPolicy {
    template<typename T>
    static consteval bool accept() {
        return boost::typelayout::is_byte_copy_safe_v<std::remove_cv_t<T>>();
    }
};
```

XOffset can delete:
- `accept_all_members_impl` (duplicates TypeLayout P2996 traversal)
- `accept_all_bases_impl` (duplicates TypeLayout P2996 traversal)
- `opaque_element_types<T>` trait and all its specializations (replaced by `opaque_elements_safe`)
- All `XOFFSET_REGISTER_*` macro element-check clauses

XOffset retains:
- `StrictPolicy` (adds signature comparison on top of byte_copy_safe)
- Custom policies (e.g. SmallTypePolicy)
- `diagnose_unsafe_members` (precise error location)
- `is_polymorphic` fast-path rejection (optional optimization, byte_copy_safe handles it correctly but slower)
- Migration strategy binding (XCompactor::migrate_as) -- orthogonal concern

### No breaking changes

- `is_local_serialization_free_v<T>` unchanged
- All existing signatures unchanged
- All existing tests pass

## Design Rationale

### Why not just make `is_local_serialization_free` recursive?

`is_local_serialization_free` answers a specific formal question: "does T satisfy C++ trivially_copyable AND have no pointers?" This is a **language-level** property with precise semantics. Making it recursive would change its meaning: a struct containing XString would suddenly be "serialization free" even though `std::is_trivially_copyable_v<that struct>` is false. That would violate the established semantics.

`is_byte_copy_safe` is a separate, broader concept: "given the registered opaque guarantees, is T safe for byte-copy?" It subsumes serialization-free but has weaker formal guarantees (depends on user-provided opaque annotations being correct).

### The two predicates form a hierarchy

```
is_local_serialization_free_v<T> == true
    ==> is_byte_copy_safe_v<T> == true      (always)

is_byte_copy_safe_v<T> == true
    =/=> is_local_serialization_free_v<T>    (not necessarily)
```

`serialization_free` is the strict C++ guarantee. `byte_copy_safe` is the pragmatic guarantee with user-annotated extensions. Both are useful; neither replaces the other.

### Why put the recursion in TypeLayout, not leave it in XOffset?

1. **TypeLayout already has the P2996 infrastructure** -- `get_member_count`, `get_base_count`, `nonstatic_data_members_of`, `bases_of` traversals in `reflect.hpp` and `layout_traits.hpp`.
2. **Any consumer of TypeLayout's opaque system needs this recursion** -- XOffset is the first, but any shared-memory / IPC framework would need the same check.
3. **Eliminates version drift** -- XOffset's external TypeLayout copy has already diverged. Putting the check in TypeLayout means one source of truth.
4. **TypeLayout owns the opaque contract** -- It defines `has_opaque_signature`, `pointer_free`, the RELOCATABLE macros. The recursive check is a natural closure of that contract.

## Open Questions

1. **Naming**: `is_byte_copy_safe` vs `is_relocatable_safe` vs `is_opaque_domain_safe`? The name should convey "memcpy safe given opaque guarantees" without implying C++ trivial copyability.

2. **Polymorphic fast-path**: Should `is_byte_copy_safe` include `!is_polymorphic_v<T>` as a fast rejection? XOffset does this. It's correct (polymorphic types have vptrs = pointers) and faster than waiting for the signature scan to find `ptr[`. Trade-off: adds a non-reflection check to a reflection-based function.

3. **Union handling**: `is_local_serialization_free` accepts unions (if trivially_copyable + no pointer). Should `is_byte_copy_safe` recurse into union members? XOffset currently rejects unions at Branch 4. TypeLayout's `compute_has_padding` skips unions ("padding is semantically ambiguous"). Suggest: accept unions only if `is_local_serialization_free` (no recursion into union arms for opaque members, as the active member is runtime-determined).

4. **`opaque_elements_safe` default**: For `TYPELAYOUT_OPAQUE_TYPE_RELOCATABLE` (no element types), the default is `true`. For `TYPELAYOUT_REGISTER_OPAQUE` (trivially_copyable required), the default should also be `true`. Should the default trait return `true` or `false`? Suggest: `false` (conservative), with explicit `true` generated by each macro.
