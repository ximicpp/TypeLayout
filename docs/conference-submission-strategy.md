# 会议投稿经验与 2026 策略

## 结论

2025 年 `XOffsetDatastructure` 这次未被录用，主因不是题目没有价值，而是多个会在主会场评审里持续失分的问题叠加在一起：`outline` 写成了接近 slide deck 的长文、标题和摘要过长且术语先行、60 分钟内容明显过载、与往年内容区分度不足、以及整份稿件更像“方案/库介绍”而不是“观众听完能学会什么 C++ 方法”。

这份文档的目标不是复述遗憾，而是把这次失败转成以后再投主会场类技术大会时可直接执行的策略。

## Case Summary

### 2025 投稿结果

- 总投稿数：243
- 投稿人：163
- 总 review 数：5700+
- 相对评分区间：`59` 到 `-79`
- 本次得分：`-7`

### 评审反馈的高信号归纳

- `outline` 太长，像 slide deck，不像评审用的提案大纲
- 很多评审不会完整读完这么长的文档
- 内容量过大，60 分钟很难讲清楚
- 标题和描述太长
- 题目与去年同类大会的相近内容重复感很强
- 即使题目本身有价值，呈现方式也会让评审担心现场是否好跟

### 对原稿的客观观察

原始 2025 文稿不是一份“短 outline”，而是一份约 `2564` 词、包含 8 个一级章节、大量二三级小节、代码块、性能 benchmark、市场分析、方案架构、实现细节和扩展方向的长文。对于主会场评审流程来说，这种写法天然会制造三个问题：

- 评审很难在短时间内抓到核心价值
- 评审会先担心“这是不是根本讲不完”
- 评审会更容易把它当成完整课程/白皮书，而不是一个一小时 talk

## Root Causes

### 1. `outline` 写成了 slide deck

**问题**

- 提案不只是“信息多”，而是结构上已经像完整讲稿
- 一级到三级标题过多，章节推进方式像在按页讲内容
- 评审反馈里直接出现了 “That is not an outline, that is a slide deck”

**为什么会失分**

- 主会场评审首先要快速判断：主题是什么、听众为什么该来、这 60 分钟具体讲什么
- 一旦 `outline` 太长，评审不但读不完，还会把“内容过多”直接映射成“speaker 讲不完 / 讲不清”

**对应策略**

- `outline` 只保留 4 到 6 个高层段落
- 每段最多 3 个要点，只写“这段讲什么”，不写“这段怎么讲完”
- 不在提案里堆代码块、benchmark 表、市场分析和实现全景

### 2. 标题与摘要过长且术语先行

**问题**

- 标题堆叠了方案名、技术术语和目标结果
- 摘要第一句从方案/库本身切入，而不是从听众熟悉的问题切入
- `XOffsetDatastructure`、`Zero-encoding/Zero-decoding`、`Compile-time Type Signatures` 这些词对非本领域评审门槛较高

**为什么会失分**

- 标题和摘要前两句决定评审是否愿意继续读
- 如果读者先看到的是陌生概念，而不是问题和收益，就更容易把提案归类为 niche 或产品宣传

**对应策略**

- 标题先讲问题和收益，再决定是否需要带机制名
- 摘要前两句必须独立成立，即使拿掉项目名也能讲清楚 what / who / why
- `问题先于机制`，`方法先于项目名`

### 3. 60 分钟内容严重过载

**问题**

- 原稿同时试图覆盖库背景、行业动机、性能 benchmark、平台假设、类型子集设计、Boost container 选型、动态类型系统、compile-time string、signature 生成、runtime verify、应用案例、未来方向

**为什么会失分**

- 评审不只看内容是否有趣，也看内容是否能在现场被顺畅吸收
- 主会场评审通常都非常敏感于“太多 material 挤进 60 分钟”
- 一旦评审开始担心“讲不完”，即使题目本身不错，也会倾向降分

**对应策略**

- 一个主会场 talk 只保留一条主线和 3 个以内强例子
- 所有“可删但想讲”的内容默认删掉
- benchmark、实现细节、边缘扩展、市场论证，只能服务主线，不能各自形成副主题

### 4. 与前一年内容区分度不足

**问题**

- 反馈中有多条明确指出“去年已经有类似甚至同样的 XOffsetDatastructure talk”
- 评审不仅感到相似，还提到去年的版本现场效果一般

**为什么会失分**

- 如果 committee 认为这是“去年的 talk 再来一次”，那就不是在和别的提案公平竞争，而是在先解决一个额外问题：为什么值得再收一遍
- 只要“新意”不够显著，评审就会更保守

**对应策略**

- 每次再投前先回答一个硬问题：与上一版相比，新的核心价值到底是什么
- 如果只是“补了一些实现细节”或“内容更完整了”，通常不够
- 必须在标题、摘要和 `outline` 里直接写出本次新增的角度、方法或受众价值

### 5. 更像产品/方案介绍，而不是观众收益导向

**问题**

- 原稿的叙事中心是 “XOffsetDatastructure 是什么、为什么快、怎么实现”
- 观众真正带走的东西被放在后面，且被大量方案细节覆盖

**为什么会失分**

- 主会场更偏好 “这能让听众更会写 C++、更会做工程判断”
- 如果整份提案读起来像某个库/框架/内部方案的宣传稿，就很难打进最强竞争区

**对应策略**

- 任何主会场提案都要能回答：听众明天能据此做什么决定
- 把“项目名”退到背景位置，把“通用方法”和“工程 workflow”放到前面
- 评审读完后应该记住问题和方法，而不是只记住一个名词

## 评审真正想看到什么

结合公开投稿建议和这次反馈，主会场评审真正想看到的是下面这些硬规则：

- `title + abstract` 前两句必须直接讲清 `what / who / why`
- 尽量 `avoid jargon`，尤其不要在最前面堆陌生术语
- 题目可以有深度，但必须先让非该子领域的评审看懂价值
- `broad appeal with deep appeal`：不是把题目讲浅，而是把入口讲宽
- 不要试图在提案里覆盖完整 material；评审更希望看到一条清晰、可讲完的主线
- 如果和前一年内容接近，必须明确证明这次有足够显著的新增价值
- 主会场更偏好“帮助听众更会做 C++ 工程判断”的 talk，而不是产品发布说明

## 2026 Strategy

2026 再投时，必须坚持以下策略，并默认把它们当作硬约束而不是建议：

### 1. 问题先于机制

- 摘要先写听众熟悉的问题，再写使用了什么机制
- 评审先感受到痛点，才会愿意听反射、签名、ABI 这些细节

### 2. 方法先于项目名

- 优先把提案写成“可迁移的方法”和“工程工作流”
- 项目名只作为 supporting detail，不做标题主角

### 3. `workflow` 先于实现细节

- 先说明“听众可以怎么做”，再解释内部怎么实现
- 优先讲：
  `如何验证`
  `如何接入 CI`
  `如何识别边界`
- 次要讲：
  `内部模板技巧`
  `辅助类型`
  `底层字符串拼装`

### 4. 边界与不承诺项要提前写清

- 不等到后半段再补充限制
- 一开始就说明这个方法证明什么、不证明什么
- 这样会提高评审对提案的信任感

### 5. `outline` 必须保持可扫读

- 评审应该在几分钟内读完 `outline`
- 任何让人产生“这像讲稿”的结构都默认删掉
- 只保留判断录用所必需的信息

### 6. 绝不破坏 `double-blind`

- 正式提交版本不出现仓库、自有项目、个人身份线索
- 避免“my library / our repository / this project from last year”一类表述

## Do / Don't

### Do

- 先写听众痛点，再写解决机制
- 标题和摘要前两句直接给出收益
- 保留 3 个以内最强例子
- 提前证明这 60 分钟可以讲完
- 把 talk 写成一种方法，而不是一个产品
- 在提案里清楚说明边界、限制和非目标
- 明确与往年内容的差异

### Don't

- 不要把 `outline` 写成讲稿或 slide deck
- 不要在标题里堆项目名和陌生术语
- 不要在 `abstract` 里塞完整 benchmark、实现全景和次要论证
- 不要试图一次覆盖“背景、方案、实现、优化、扩展、未来”
- 不要默认评审会耐心读完长文
- 不要提交“去年的 talk + 一些补充”
- 不要出现任何会破坏 `double-blind` 的措辞

## 投稿前自评工具

> 这是一个**启发式工具**，不是评审结果预测器。
> 目标是提升 `Title + Abstract + Outline` 的首轮评审可读性、主会场适配度，以及多版本提案之间的横向比较效率。

### 使用方式

适用场景：

- 单稿自查：判断“现在能不能投”
- 多稿对比：判断“哪个版本更接近中稿”

建议流程：

1. 先过一遍“**一票否决项**”
2. 再逐项填写“**评分项**”
3. 最后写“**总评模板**”

评分建议：

- 使用 `1-5` 分制
- `1-2` = 明显风险
- `3` = 可投但有明显短板
- `4` = 整体较强
- `5` = 非常接近可提交状态

总分只做相对判断，不映射成真实中稿概率。

推荐阈值：

- `36-45`：可投
- `27-35`：建议再改
- `26` 及以下：暂不建议投

### 一票否决项

命中任意一条，都应先修改再投：

- 标题过长，看完标题仍说不清“这场 talk 的新意是什么”
- 摘要前两句仍然不能直接回答“这是什么”“为什么值得听”
- `Outline` 更像 slide deck、讲稿目录或技术白皮书，而不是 `4-6` 段的演讲结构
- 明显看不出与过往 talk、已有项目演讲或历史版本的差异
- 内容范围显著超出 `60` 分钟承载能力
- 术语密度过高，非该领域评委在 1 分钟内无法建立问题背景
- 双盲匿名性明显不足，文案里充满项目指纹或特征性命名

### 总评模板

提交前至少写出这三行：

- **一句话结论**：这版是“可投 / 建议再改 / 暂不建议投”
- **最大优势**：最能打动评审的一点是什么
- **最大风险**：如果被拒，最可能死在哪个问题上

### 最小版提交前 Checklist

如果没时间完整打分，至少快速过下面这些检查：

- 标题是否能被非该子领域评审快速理解
- 摘要前两句是否不依赖背景知识也能成立
- 是否能在一句话内讲清楚这场 talk 的核心价值
- 是否能明确写出 3 个以内 `takeaway`
- `outline` 是否可以在几分钟内扫完
- 是否真的能在 `60` 分钟内讲完
- 是否有只保留最强 3 个例子
- 是否清楚说明了“证明什么、不证明什么”
- 是否与往年内容有可感知的明显差异
- 是否存在项目名、仓库名、身份信息等 `double-blind` 风险
- 是否把“方法”和“workflow”放在了“项目介绍”前面

**主会场评审要的是一场让评审一眼看出价值、让观众一小时内能吸收并带走方法的演讲提案，不是一份证明自己准备得非常充分的技术白皮书。**
